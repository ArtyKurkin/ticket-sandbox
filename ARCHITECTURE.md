# ARCHITECTURE

Этот файл описывает архитектуру Ticket Sandbox: как связаны Django, очереди, учебные задания, Docker-контейнеры, ttyd-терминал, автопроверка и ручная проверка наставником.

## Общая идея

Ticket Sandbox — учебная тикетница для практики стажеров технической поддержки.

Проект имитирует полный цикл работы с клиентским обращением:

1. Стажер открывает учебный тикет.
2. Запускает изолированное Docker-окружение.
3. Диагностирует проблему через веб-терминал.
4. Исправляет техническую проблему.
5. Отправляет техническую часть на автопроверку.
6. После успешной технической сдачи пишет ответ клиенту и внутренний комментарий, если заданию нужна ручная проверка.
7. При необходимости получает ручную проверку ответа от наставника.

Главное архитектурное разделение:

```text
check.sh проверяет техническую часть
наставник проверяет только ответ клиенту
```

Если техническая часть уже успешно пройдена, доработка текста наставником не требует повторного запуска Docker-контейнера и `check.sh`.

## Основной поток

```text
Пользователь
  ↓
Django dashboard
  ↓
TaskAttempt
  ↓
Docker task container
  ↓
ttyd terminal container
  ↓
check.sh
  ↓
CheckRun
  ↓
TaskAttempt.technical_passed_at
  ↓
ручная проверка ответа наставником, если требуется
```

## Основные сущности

### Queue

Очередь учебных заданий.

Очередь определяет:

- какие задания видит пользователь;
- где искать Docker-окружение задания;
- в каком порядке показывать задачи;
- к какому сценарию обучения относится задание.

Используемые очереди:

- `candidate` — задания для кандидатов;
- `l1` — основная очередь для стажеров, которые готовятся к работе на L1;
- `l2` — будущие более сложные задания;
- `admin` — служебная очередь.

Отдельной очереди `trainee` в проекте нет.

Стажер считается претендующим на L1 и решает задачи в очереди `l1`.

### TraineeProfile

Профиль пользователя в тренажере.

Хранит уровень пользователя:

- `candidate`;
- `l1`;
- `l2`;
- `admin`.

Профиль создается автоматически через signal при создании пользователя.

Права наставника не хранятся отдельным полем в профиле.

Наставник определяется через стандартное поле Django:

```python
User.is_staff
```

Такой подход выбран, чтобы не было рассинхрона между отдельным флагом наставника и реальными правами пользователя.

### Task

Учебное задание.

Задание принадлежит очереди через обязательное поле:

```python
Task.queue
```

`Task.queue` обязателен, потому что без очереди невозможно понять:

- кто должен видеть задание;
- где лежит Docker-окружение;
- в каком порядке показывать задачу.

Поле `Task.queue_name` удалено и больше не используется.

`Task.slug` уникален внутри очереди, но одинаковый slug может существовать в разных очередях.

Например, такие задания допустимы:

```text
candidate/nginx-not-starting
l1/nginx-not-starting
```

Путь к Docker-окружению строится по схеме:

```text
training_tasks/<queue_slug>/<task_slug>
```

Важное поле для ручной проверки:

```python
Task.requires_manual_review
```

Если `requires_manual_review = True`, после успешной технической проверки ответ клиенту должен проверить наставник.

Если `requires_manual_review = False`, достаточно успешной технической проверки.

### TaskAttempt

Попытка прохождения задания конкретным пользователем.

Именно `TaskAttempt`, а не `Task`, хранит состояние работы:

- статус;
- ответ клиенту;
- внутренний комментарий;
- количество проверок;
- количество перезапусков;
- имя task-контейнера;
- имя terminal-контейнера;
- порт терминала;
- ссылку на терминал;
- команду подключения;
- номер попытки `attempt_number`;
- признак текущей попытки `is_current`;
- вывод последней проверки;
- дату успешной технической проверки;
- комментарий наставника;
- решение наставника;
- кто и когда проверил ответ;
- когда стажер увидел комментарий наставника.

Одно и то же задание могут проходить разные пользователи, и у каждого должна быть своя попытка и свой контейнер.

Ключевое поле:

```python
TaskAttempt.technical_passed_at
```

Оно фиксирует, что техническая часть задания успешно выполнена.

Если `technical_passed_at` заполнено:

- техническая часть считается выполненной;
- стажер может переходить к следующему заданию;
- наставник проверяет только текст ответа;
- при доработке текста Docker и `check.sh` повторно не нужны.

### CheckRun

`CheckRun` хранит историю запусков автопроверки.

Каждый запуск `check.sh` создает отдельную запись.

Это нужно, чтобы видеть не только последний результат, но и всю историю попыток проверки.

`CheckRun` хранит:

- связанную `TaskAttempt`;
- время запуска;
- результат проверки;
- exit code;
- вывод проверки.

История проверок отображается на странице тикета и доступна в админке.

Через inline в админке ручное создание `CheckRun` запрещено: история должна создаваться системой при запуске проверки.

## Доступы к очередям

Текущая схема:

```text
candidate → candidate
l1        → l1
l2        → l1, l2
admin     → candidate, l1, l2, admin
```

Наставники и администраторы, у которых `User.is_staff = True`, попадают в mentor dashboard.

Обычные пользователи попадают в trainee dashboard.

## Последовательность заданий

Задания внутри очереди идут по полю `order`.

Первая задача доступна сразу.

Следующая задача становится доступной только после того, как предыдущая задача в этой же очереди была успешно пройдена технически.

Для этого используется не только общий статус, но и факт заполнения:

```python
TaskAttempt.technical_passed_at
```

Это важно, потому что ручная проверка наставника относится к качеству ответа клиенту, а не к техническому прохождению задания.

## Статусы попытки

Жизненный цикл `TaskAttempt`:

```text
new
  ↓
in_progress
  ↓
on_review / failed / passed
```

Смысл статусов:

- `new` — попытка создана, окружение еще не запущено;
- `in_progress` — контейнер запущен, стажер работает;
- `on_review` — техническая часть пройдена, ответ клиента ожидает ручной проверки наставника;
- `failed` — автопроверка не прошла или наставник отправил ответ на доработку;
- `passed` — задание принято.

При успешной автопроверке дополнительно заполняется:

```python
technical_passed_at
```

Именно это поле является надежным признаком успешной технической сдачи.

Если `technical_passed_at` заполнено, обычный запуск или перезапуск окружения не должен сбрасывать прогресс.

## Ручная проверка наставником

Ручная проверка используется для оценки ответа клиенту.

Она не заменяет и не дублирует `check.sh`.

Логика такая:

```text
check.sh успешен
  ↓
technical_passed_at заполнен
  ↓
если Task.requires_manual_review=True
  ↓
наставник проверяет client_answer
```

Наставник может:

- принять ответ;
- отправить ответ на доработку;
- оставить комментарий.

Для этого используются поля:

```python
mentor_feedback
mentor_decision
mentor_reviewed_by
mentor_reviewed_at
mentor_feedback_seen_at
```

Если наставник отправил ответ на доработку:

- стажер правит только текст;
- техническая часть не сбрасывается;
- Docker-контейнер не нужно запускать заново;
- `check.sh` не нужно запускать повторно.

## Баннер новых комментариев наставника

Если наставник оставил новый комментарий, стажер видит баннер на странице задания.

Баннер нужен, чтобы стажер не пропустил обратную связь.

После просмотра комментария заполняется:

```python
mentor_feedback_seen_at
```

После этого комментарий считается просмотренным.

## Прогресс по очереди

На дашборде стажера отображается прогресс по очереди.

Прогресс считается по задачам, которые пользователь успешно прошел технически.

Ключевой критерий:

```python
technical_passed_at is not None
```

Это позволяет отделить техническое прохождение задания от ручной проверки текста.

## Запуск задания

Когда стажер нажимает «Начать работу»:

1. Django получает `attempt_id`.
2. Проверяет, что попытка принадлежит текущему пользователю.
3. Проверяет, что задание доступно пользователю.
4. Проверяет, что техническая часть еще не закрыта через `technical_locked`.
5. Создает task-контейнер.
6. Находит свободный порт для терминала.
7. Создает terminal-контейнер с ttyd.
8. Передает ttyd base path, если включен terminal gateway.
9. Сохраняет данные контейнеров в `TaskAttempt`.
10. Переводит попытку в `in_progress`.

Упрощенно:

```text
start_task view
  ↓
create_task_container()
  ↓
get_free_port()
  ↓
create_terminal_container()
  ↓
TaskAttempt.status = in_progress
```

Если `technical_locked=True`, обычный запуск окружения запрещен, чтобы не сбросить успешную техническую сдачу.

## Docker-окружения заданий

Учебные окружения лежат в директории `training_tasks`.

Структура:

```text
training_tasks/
└── <queue_slug>/
    └── <task_slug>/
        ├── Dockerfile
        └── files/
            ├── check.sh
            └── остальные файлы окружения
```

Примеры текущих заданий:

```text
training_tasks/l1/nginx_not_starting/
training_tasks/l1/sajt-ne-rabotaet-posle-perenosa/
```

Для задания в базе должны совпадать:

```text
queue.slug = l1
task.slug = <название папки задания>
```

Docker-сервис собирает путь к заданию так:

```text
training_tasks/<queue_slug>/<task_slug>
```


## task.json и sync_training_tasks

`training_tasks` — файловый источник учебных заданий.

Для каждого задания рядом с `Dockerfile` и файлами окружения должен лежать `task.json`.

Именно `task.json` считается источником правды для полей задания:

- название;
- тема тикета;
- сообщение клиента;
- имя и email клиента;
- приоритет;
- порядок;
- активность;
- необходимость ручной проверки.

Команда синхронизации:

```bash
python manage.py sync_training_tasks --strict
```

создает и обновляет записи `Task` в БД на основе структуры:

```text
training_tasks/<queue_slug>/<task_slug>/task.json
```

Перед применением изменений нужно запустить strict dry-run:

```bash
python manage.py sync_training_tasks --dry-run --strict
```

Архитектурное правило: постоянные правки учебного задания делаются через `task.json`, а не через Django admin.

Django admin остается удобным интерфейсом для просмотра, сортировки, фильтров и быстрых массовых действий. Но если изменить поля задания только в admin, следующий запуск `sync_training_tasks` может вернуть значения из файлов.

На staging команда `sync_training_tasks --strict` выполняется в CD после миграций, до сборки Docker-образов и до `collectstatic`/рестарта приложения.

## task-контейнер

Task-контейнер — это контейнер с учебной проблемой.

Внутри него стажер выполняет диагностику и исправляет поломку.

Контейнер должен быть одноразовым: его можно удалить и создать заново при перезапуске задания.

## terminal-контейнер

Terminal-контейнер запускает ttyd и дает веб-доступ к shell внутри task-контейнера.

Terminal-контейнер публикует ttyd только на localhost:

```python
ports = {"7681/tcp": ("127.0.0.1", port)}
```

Это запрещает прямой внешний доступ к случайному ttyd-порту.

Basic Auth для ttyd не используется.

Доступ к терминалу контролируется через terminal gateway:

```text
nginx /terminal/<attempt_id>/<port>/
  ↓
auth_request /_terminal_auth
  ↓
Django /terminal-auth/
  ↓
204 разрешает proxy_pass на 127.0.0.1:<port>
401/403 запрещает доступ
```

Такой подход позволяет оставить ttyd простым техническим контейнером, а авторизацию выполнять на уровне Django-сессии.

## Terminal gateway

Terminal gateway включается переменной:

```env
TERMINAL_GATEWAY_ENABLED=true
```

При включенном gateway `TaskAttempt.terminal_url` строится не как прямой URL на порт, а как внутренний путь приложения:

```text
/terminal/<attempt_id>/<port>/
```

Запросы на этот путь принимает nginx.

Перед проксированием WebSocket-соединения nginx выполняет `auth_request`:

```nginx
auth_request /_terminal_auth;
```

Внутренний location `/_terminal_auth` обращается к Django:

```text
/terminal-auth/
```

Так как auth subrequest приходит на отдельный URL без параметров, nginx передает исходный terminal URI в заголовке:

```text
X-Original-URI
```

Django view `terminal_auth` разбирает этот URI и проверяет:

- пользователь авторизован;
- попытка существует;
- пользователь является владельцем попытки или имеет `User.is_staff=True`;
- запрошенный порт совпадает с `TaskAttempt.terminal_port`;
- terminal-контейнер существует;
- `terminal_url` не пустой;
- попытка не закрыта технически;
- попытка доступна пользователю.

Результаты проверки:

```text
204 → доступ разрешен, nginx проксирует терминал
401 → пользователь не авторизован
403 → доступ запрещен
```

WebSocket проксируется через nginx с заголовками:

```nginx
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;
```

`proxy_pass` для terminal не должен заканчиваться слешем:

```nginx
proxy_pass http://127.0.0.1:$terminal_port;
```

`Host` передается как `$http_host`, чтобы локальный вход через `localhost:8081` не ломался по CSRF.

Конфиги:

```text
deploy/nginx/ticket-sandbox-local.conf
deploy/nginx/ticket-sandbox.conf.example
```

Terminal-auth логирует причины отказов через логгер:

```text
sandbox.terminal
```

Примеры причин:

```text
anonymous
invalid_original_uri
attempt_not_found
forbidden_user
wrong_port
missing_terminal_container
empty_terminal_url
technically_locked
attempt_not_available
```

Если наставник с `User.is_staff=True` открывает терминал стажера, это логируется отдельным audit-событием:

```text
mentor_terminal_access
```

Это не ошибка, но чувствительное действие, поэтому оно пишется на уровне `WARNING`.

## Автопроверка

Автопроверка отвечает только за техническую часть задания.

Когда стажер запускает проверку до технической сдачи:

1. Django проверяет, что попытка принадлежит пользователю.
2. Проверяет, что попытка доступна пользователю.
3. Проверяет, что попытка не находится в финальном `passed` или уже в `on_review`.
4. Проверяет, что у попытки есть запущенный task-контейнер.
5. Запускает `check.sh` внутри task-контейнера.
6. Создает запись `CheckRun`.
7. Если проверка успешна, заполняет `technical_passed_at`.
8. Если проверка успешна, удаляет terminal-контейнер и task-контейнер.
9. Если ручная проверка не нужна, переводит попытку в `passed`.
10. Если ручная проверка нужна и это зачётная попытка, оставляет попытку в `in_progress`, чтобы стажер заполнил ответ клиенту и внутренний комментарий.
11. Если проверка неуспешна, переводит попытку в `failed`.

После успешной технической сдачи для задания с `requires_manual_review=True` стажер отправляет уже не Docker-проверку, а текст на ручную проверку:

1. Django проверяет, что `technical_passed_at` уже заполнен.
2. Проверяет, что заполнены `client_answer` и `trainee_report`.
3. Сохраняет текст ответа и внутренний комментарий.
4. Переводит попытку в `on_review`.
5. Через `transaction.on_commit(...)` отправляет Telegram-уведомление наставникам.

Упрощенно:

```text
check_task view
  ↓
check_task_container()
  ↓
check.sh
  ↓
CheckRun
  ↓
exit code 0 → technical_passed_at
  ↓
requires_manual_review=False или extra attempt → passed
requires_manual_review=True + credit attempt → in_progress, форма ответа открыта
  ↓
стажер отправляет текст
  ↓
on_review
  ↓
ручная проверка наставником
```

Вывод `check.sh` сохраняется в `CheckRun.output` и `TaskAttempt.last_check_output`.

В application logs вывод `check.sh` не пишется, чтобы не раздувать логи и случайно не отправлять туда лишние данные.

## Перезапуск задания

Перезапуск нужен, если стажер сломал окружение или зашел в тупик до успешной технической сдачи.

При обычном перезапуске:

1. Удаляется старый terminal-контейнер.
2. Удаляется старый task-контейнер.
3. Создается новый task-контейнер.
4. Создается новый terminal-контейнер.
5. Увеличивается `restart_count`.
6. Сбрасывается `finished_at`.
7. Попытка возвращается в `in_progress`.

Обычный перезапуск запрещен, если техническая часть уже успешно сдана:

```text
technical_passed_at заполнен
  ↓
technical_locked=True
  ↓
обычный restart/start запрещен
```

Это нужно, чтобы случайно не удалить успешный результат и не откатить прогресс по очереди.

## Повторные тренировочные попытки

Если стажер хочет пройти задание заново после успешной технической сдачи, используется отдельное осознанное действие.

В этом случае создается новая `TaskAttempt`:

```text
attempt_number > 1
is_current = True
```

Предыдущая успешная попытка сохраняется в истории.

Дополнительная попытка считается тренировочной:

```text
is_extra_attempt=True
is_credit_attempt=False
```

Последствия:

- тренировочная попытка не откатывает прогресс по очереди;
- тренировочная попытка не заменяет зачётную попытку;
- тренировочная попытка не отправляется наставнику на ручную проверку;
- mentor dashboard работает только по зачётным попыткам `attempt_number=1`.

## Исторические попытки

Когда создается новая тренировочная попытка, предыдущая попытка остается в истории:

```text
is_current = False
```

Историческая попытка доступна только для просмотра.

Для нее недоступны:

- запуск окружения;
- перезапуск окружения;
- отправка на проверку;
- открытие web-терминала;
- staff-команда подключения к контейнеру.

Backend дополнительно блокирует action-view для исторических попыток, чтобы нельзя было выполнить действие прямым POST-запросом.

## Healthcheck

В проекте есть простой endpoint для проверки доступности приложения:

```text
/healthz/
```

Он возвращает JSON:

```json
{"status": "ok", "service": "ticket-sandbox"}
```

`/healthz/` используется в staging smoke-check после деплоя.

Endpoint не заменяет полноценный мониторинг, но позволяет быстро проверить, что Django-приложение отвечает после рестарта.

## Telegram-уведомления

Telegram-уведомления используются как быстрый операционный канал для наставников.

Интеграция работает через два слоя:

```text
sandbox/services/telegram.py       # низкоуровневая отправка сообщения
sandbox/services/notifications.py  # бизнес-события тренажера
```

Низкоуровневый сервис читает переменные окружения:

```env
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
```

Если токен или chat id не заданы, уведомления считаются выключенными.
Основная логика тренажера при этом не ломается.

Если Telegram API недоступен, ошибка логируется через логгер:

```text
sandbox.telegram
```

Пользовательский сценарий не должен падать из-за проблем с Telegram.

Текущие события:

1. Попытка отправлена наставнику на ручную проверку.
2. Стажер технически прошел все активные задания в доступных очередях.

Уведомления вызываются после сохранения изменений через:

```python
transaction.on_commit(...)
```

Это нужно, чтобы сообщение ушло только после успешной записи состояния в БД.

Для ручной проверки уведомление отправляется не сразу после `check.sh`, а после того, как стажер заполнил ответ клиенту и внутренний комментарий, и попытка реально перешла в `on_review`.

## Mentor dashboard

Mentor dashboard доступен пользователям с:

```python
User.is_staff = True
```

В dashboard наставника есть:

- статистика по попыткам;
- фильтры;
- список попыток;
- информация об очереди;
- статус попытки;
- количество проверок;
- количество перезапусков;
- последний результат автопроверки;
- ручная проверка ответа клиенту;
- бейдж «Ждут проверки».

Бейдж «Ждут проверки» считается отдельно от текущих фильтров dashboard.

Он показывает количество зачётных текущих попыток, которые реально ожидают ручной проверки:

Это нужно, чтобы наставник сразу видел общий объем ожидающих проверок, даже если сейчас в таблице включен фильтр по другой очереди или статусу.

Наставник может оставить фидбек и решение по ответу стажера.

## Timeout автопроверки и ошибки Docker API

Автопроверка `check.sh` запускается внутри task-контейнера через команду `timeout`.

Время выполнения ограничивается настройкой:

```env
CHECK_TASK_TIMEOUT_SECONDS=60
```

Если `check.sh` выполняется дольше указанного времени, проверка завершается с ошибкой, а в вывод добавляется понятное сообщение для стажера и наставника.

Базовая обработка ошибок Docker API добавлена для основных action-view:

```text
start_task
restart_task
check_task
```

Если Docker API недоступен или операция завершается исключением:

- пользователь не получает 500;
- попытка переводится в `failed`;
- причина сохраняется в `TaskAttempt.last_check_output`;
- ошибка логируется через `sandbox.terminal`;
- пользователь возвращается на страницу тикета с понятным flash-сообщением.

Это не заменяет будущий Celery + Redis, но делает MVP устойчивее перед первым staging-деплоем.

## Ограничения текущей архитектуры

Текущая архитектура подходит для MVP и staging-пилота, но перед production-использованием нужно закрыть несколько важных зон.

### Docker-операции синхронные

Сейчас запуск, проверка и перезапуск контейнеров выполняются синхронно из Django view.

При нескольких стажерах это может блокировать Django-процессы.

Будущее решение:

```text
Django view
  ↓
Celery task
  ↓
Docker operation
  ↓
TaskAttempt update
```

Для этого понадобится Celery + Redis.

### Нужен polling статуса проверки

Сейчас пользовательский запрос может ждать завершения долгой Docker-операции или `check.sh`.

Следующий UX-шаг — запускать проверку и показывать живой статус:

```text
checking
  ↓
passed / failed / on_review
```

Даже без Celery это можно сделать промежуточно через фоновый поток и polling endpoint, а позже заменить на полноценную очередь задач.

### Terminal gateway требует production hardening

Terminal gateway уже проверяет доступ через Django-сессию и nginx `auth_request`.

Перед production-использованием нужно отдельно проверить:

- TLS;
- secure cookies;
- CSRF-настройки;
- корректные `ALLOWED_HOSTS`;
- заголовки `X-Forwarded-*`;
- ограничения на размер и время соединений;
- логи nginx и Django;
- поведение при перезапуске контейнеров и Django.

### Нужна асинхронная модель статусов

Сейчас пользовательский запрос сразу запускает Docker-операцию.

После выноса в Celery понадобится добавить промежуточные состояния:

```text
starting
checking
restarting
cleanup_failed
```

Это позволит корректно отображать долгие операции в интерфейсе.
